Tổng quan

QuraEx Backend là gì?

Backend microservices cho nền tảng QuraEx — quản lý user story, test artifact, sinh test bằng AI, thực thi và tích hợp.

Nền tảng

.NET 10 · Vertical Slice + CQRS/MediatR

Giao tiếp

Event qua MassTransit / RabbitMQ

Cổng vào

Kong API Gateway (DB-less) — định tuyến + xác thực JWT ở biên

Dữ liệu & dev

PostgreSQL (1 DB/service), Redis, điều phối local bằng .NET Aspire
Kiến trúc

8 service theo nghiệp vụ

Client → Gateway (Kong DB-less) → từng service độc lập, mỗi service có DB riêng và phát/nhận event qua RabbitMQ.

Client / FE→ Kong Gateway→ Identity Workspace Authoring Test-Artifact
AI-Generation Execution Integration Notification ⇄RabbitMQ (events)

Mỗi service = 4 lớp: Api · Domain · Infrastructure · Contracts (event công khai) + bộ Tests.

Quyết định kiến trúc

3 lựa chọn cần hiểu (hay bị hỏi)

① Monorepo — KHÔNG tách mỗi service 1 repo

Microservice = độc lập lúc deploy/runtime, không phải số lượng repo. Tách repo lúc này = gánh versioning, CI ×8, mất Aspire F5 — chưa có lợi ích khi đội nhỏ.

② GitHub Organization QuraEx

Chứa repo BE (và FE sắp dựng). Cho namespace package, teams/RBAC, secret dùng chung — độc lập với chuyện tách repo.

③ Database-per-service + Outbox pattern

Mỗi service 1 DB riêng, không share. Nhất quán dữ liệu qua event + transactional outbox, không cần distributed transaction.

Hay bị hỏi

IdP & AWS — đừng nhầm 2 thứ

"AWS IAM" ≠ login user

AWS IAM = phân quyền tài nguyên cloud (pod nào đọc S3, gọi RDS — chuyện deploy/CI). Login user = OpenIddict. Hai thứ khác nhau hoàn toàn.

Vì sao OpenIddict, không Cognito?

Chạy được offline/local (Aspire/Docker) — Cognito bắt mỗi dev có AWS cred ngày đầu. Tự dựng OIDC = khoe năng lực thật.

Federation-ready

Gateway Kong validate JWT qua authority/JWKS cấu hình được. Lên AWS có thể federate Cognito mà không sửa business code — chỉ đổi 1 biến config.

AWS IAM thật dùng ở đâu?

IAM Role cho EKS/ECS (IRSA) · quyền Execution ghi S3 (thay MinIO ở prod) · Secrets Manager giữ secret.

Trái tim hệ thống

Luồng "Generate test case" end-to-end

Một luồng chạm mọi pattern: gRPC · Saga · Outbox · CQRS · Idempotency · SignalR · read-model.

  • 1Authoring: user viết story → bấm Generate.
  • 2AI Gen: tạo job, trả 202 + jobId ngay; client theo dõi qua Redis/SignalR.
  • 3AI Gen → Authoring: gRPC lấy nội dung story (sync vì cần ngay).
  • 4AI Gen → LLM router: OpenAI/Gemini (fallback + circuit breaker).
  • 5AI Gen: 1 transaction → job=COMPLETED + ghi TestCasesGenerated vào Outbox (giải dual-write).
  • 6TestArtifact: consume → lưu test case idempotent (processed_messages) → emit TestCasesSaved.
  • 7Song song: Authoring cập nhật story status · Integration đẩy Jira nếu có link.
  • 8Execution: chạy Playwright → artifact (MinIO/S3) → TestRunCompleted → Notification báo owner.

Luồng A · B

Đăng nhập & Mời thành viên

A · Đăng ký / đăng nhập

Identity tạo user, hash Argon2id, phát RS256 JWT (OpenIddict). Phát UserRegistered → Workspace seed profile. Kong + mọi service validate JWT qua JWKS.

B · Mời thành viên (Saga)

Workspace: Pending → AwaitingResponse → Accepted (hoặc Expired sau 7 ngày → bù trừ: hủy + báo mail). Token lưu hash. Phát MembershipChanged → Notification báo người mời.

Pattern: Outbox · idempotent consumer · Saga + compensation.

Luồng C

Viết story & import từ Jira (ACL)

Authoring→ UserStoryCreated→ TestArtifact (placeholder) Integration (link Jira)
Jira webhook→ Integration (ACL)→ dịch payload sạch→ Authoring (external_ref)

Điểm vàng: Authoring KHÔNG bao giờ biết cấu trúc Jira — Integration (ACL) dịch sạch trước. Đổi provider khác chỉ sửa Integration.

Luồng E

Thực thi test tự động (Playwright)

TestArtifact→ TestRunRequested→ Execution→ MinIO/S3 (artifact)→ TestRunCompleted→ Notification
  • Browser chạy trong container cô lập (giới hạn CPU/RAM/timeout) — bảo mật bắt buộc.
  • DB chỉ lưu đường dẫn artifact; file (screenshot/video/trace) ở object storage.
  • Case FAIL → tạo defect, có thể đẩy Jira — khép kín vòng QA.
Luồng F · chốt

Thông báo & 5 câu chốt khi trình bày

F · Notification (fan-out)

Consume TestRunCompleted / MembershipChanged → đọc notification_preferences → lưu Mongo → gửi in-app/email. Pure consumer, làm sau cùng.

  • Microservice = độc lập lúc deploy, không phải nhiều repo.
  • Mỗi service 1 DB, không đọc chéo → cần data người khác thì dùng snapshot.
  • Outbox + processed_message khắp nơi = nhất quán không cần distributed transaction.
  • gRPC hiếm, event là mặc định.
  • Eventual consistency: trễ nhẹ, đổi lại chịu lỗi tốt + scale độc lập.
Vòng đời · Saga

State: Saga mời thành viên

Pending→ AwaitingResponse→ Accepted ✅

Từ AwaitingResponse còn 3 nhánh kết thúc khác:

↳ Expired (quá 7 ngày) Declined Revoked

Token an toàn

Lưu hash; accept thì hash lại để đối chiếu — rò DB cũng không dùng được token.

Expired = bù trừ tự động

Saga timeout 7 ngày tự hủy invitation + gửi mail báo hết hạn. Không thao tác tay.

Chỉ Accepted mới tạo project_member + phát MembershipChanged lần 2 (để các service cập nhật snapshot).

Vòng đời · Job

State: generation_job (AI Gen)

PENDING→ FETCHING_STORY→ GENERATING→ PERSISTING→ COMPLETED ✅
Trong GENERATING (router LLM): Source 1→ lỗi/timeout → Source 2↛ cả 2 lỗi → FAILED ❌
  • Client theo dõi qua Redis/SignalR: PENDING→GENERATING→COMPLETED realtime, không refresh.
  • 3 mức lỗi khác nhau: lấy story lỗi (chưa side-effect → FAILED gọn) · LLM lỗi (fallback nguồn 2) · lưu TestArtifact lỗi (redeliver + idempotent tự lành).
  • COMPLETED chỉ đạt khi Outbox đã ghi cùng transaction → không bao giờ "job xong mà event mất".
